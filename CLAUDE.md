# CLAUDE.md - 多智能体协作配置

---

## 三个工作流

本配置包含三个独立的工作流：

| 工作流 | 用途 | 触发方式 | 智能体数量 |
|--------|------|----------|------------|
| **编码工作流** | 通用软件开发 | "帮我写一个XXX功能" | 3个 |
| **PPT工作流** | 专业PPT制作 | "帮我做一个PPT" | 7个 |
| **论文工作流** | 学术论文撰写（学术论文/数学建模） | "帮我写一篇论文" | 8-11个（按类型） |

---

## 工作流1: 编码工作流

> 用于通用软件开发任务

### 智能体角色

| 角色 | 职责 | 配置文件 |
|------|------|----------|
| 主智能体 (Orchestrator) | 任务协调、拆解、分配 | `orchestrator.md` |
| 编码智能体 (Coder) | 编写代码 | `coder.md` |
| 评审智能体 (Reviewer) | 评审代码质量 | `reviewer.md` |

### 工作流程
```
1. 主智能体接收任务
2. 主智能体拆解任务为子任务
3. 对于每个子任务：
   a. 调用全新的编码智能体（无上下文）
   b. 编码智能体完成任务，返回文件路径
   c. 调用全新的评审智能体（无上下文）
   d. 评审智能体评审完成，返回评审结果文件路径
   e. 如果评审失败（最多3轮）：
      - 将评审结果文件路径传回编码智能体
      - 编码智能体修复后再次提交评审
4. 主智能体整合结果，推进下一个子任务
```

### 调用方式
```
帮我写一个 [功能描述]
```

### 调用模板

#### 调用编码智能体
```
请调用 Agent 工具生成一个新的 coder 子智能体，执行以下任务：

任务描述：[具体任务内容]
输入文件：[相关文件路径]
输出要求：只返回生成的文件路径，不要返回其他信息
```

#### 调用评审智能体
```
请调用 Agent 工具生成一个新的 reviewer 子智能体，评审以下代码：

代码文件：[文件路径]
评审标准：[评审要求]
输出要求：将评审结果写入文件，只返回文件路径
```

### ⛔ 素材库验证（阶段5.5 - 强制执行）

**在调用设计评审师之前，你必须用 Grep 工具检查生成的脚本文件是否使用了素材库组件。**

检查命令（用 Grep 工具执行）：
```
pattern: "from ppt_builder import PPTBuilder|PPTBuilder|from html_ppt_builder import HTMLPPTBuilder|HTMLPPTBuilder"
file_path: [生成的脚本文件路径]
```

**如果匹配数为 0（脚本没有使用PPTBuilder）：**
1. 读取已生成的脚本文件内容
2. 重新调用 ppt_coder，但这次把现有代码传入，并追加以下指令：

```
你上次生成的PPT脚本没有使用PPTBuilder API。以下是现有代码：
[插入现有代码]

请你用 PPTBuilder 重写这个脚本：
import sys, os
sys.path.insert(0, os.path.expanduser("~/.claude/agents/assets/components"))
from ppt_builder import PPTBuilder

p = PPTBuilder("主题")
p.cover("标题", "副标题")
p.page("章节名")
p.cards("标题", [("卡片1","内容1"), ...])
p.stats("标题", [("数字","标签"), ...])
p.ending("感谢聆听")
p.save("output.pptx")

禁止手动创建形状。禁止手动定义颜色。只用PPTBuilder的方法。

注意：HTMLPPTBuilder与PPTBuilder共享相同API，page_N(p)函数同时兼容两种Builder。

现有脚本文件：[文件路径]
```

**如果匹配数 > 0（脚本使用了素材库）：正常进入阶段6。**

**最多重试2次。如果2次重试后仍未使用素材库，跳过此检查进入阶段6，但在评审报告中标注。**

=== 回退通道（非单向，支持反馈修正）===

阶段5 → 阶段4（大纲修正）：
  内容写作智能体发现大纲某节逻辑不通/结构不合理
  → 输出 rollback_feedback.md（问题描述 + 修改建议）
  → 回退到阶段4，架构设计师读取反馈 + 原大纲，输出 outline_v2.md
  → 回退点之后的已写章节作废，从回退点重新逐章写作

阶段5 → 阶段2（文献补搜）：
  内容写作智能体发现某论点缺乏文献支撑 / 引用数量不足
  → 输出 rollback_feedback.md（缺失主题 + 关键词）
  → 回退到阶段2，文献搜索智能体定向补搜
  → 阶段3自动重跑（更新综述和BibTeX）
  → 阶段4自动重跑（大纲可能微调）
  → 阶段5从回退点重新逐章写作

阶段4 → 阶段2（文献补搜）：
  架构设计师发现文献综述不足以支撑某章节设计
  → 输出 rollback_feedback.md
  → 回退到阶段2，补搜后阶段3重跑，再回阶段4

### 人工审核断点检查

> 主智能体在每个阶段完成后必须检查是否命中用户设定的断点。

#### 检查规则

在以下时机，主智能体必须读取 `requirements.md` 中的"人工审核断点"字段，判断是否需要暂停：

| 检查时机 | 对应断点 | 暂停时输出 |
|----------|----------|------------|
| 阶段2（文献搜索）完成后 | `review_literature` | 搜索到的文献列表 + 覆盖领域统计 |
| 阶段4（大纲设计）完成后 | `review_outline` | outline.md 路径 + 章节结构摘要 |
| 阶段5每章写作完成后 | `review_per_chapter` | chapter_{N}.docx 路径 + 章节摘要 |
| 阶段5全部章节完成后 | `review_draft` | 所有章节文件路径 + 全文摘要 |

#### 暂停时的行为

```
📌 人工审核断点已触发：[断点名称]

当前成果：
- [文件路径列表]

请审核后回复：
- "继续" / "通过" → 推进到下一阶段
- 修改意见（如"第二章论证不足"）→ 根据意见局部返修或触发回退
- "需要补充XX文献" → 触发阶段2回退
```

#### 断点与回退的关系

- 断点暂停期间用户给出的修改意见，优先级高于自动回退检测
- 用户可以主动要求回退到任意 earlier 阶段（不受回退次数限制）
- 用户确认"通过"后，该断点本轮不再触发（即使后续有回退重跑到该阶段）

### 阶段回退机制

> 论文写作不是流水线，内容写作阶段发现上游问题时必须支持回退修正。

#### 回退触发条件

| 触发阶段 | 回退目标 | 触发条件 | 示例 |
|----------|----------|----------|------|
| 阶段5 → 阶段4 | 大纲修正 | 大纲某节逻辑不通、结构不合理、论证链条断裂 | "第三章的因果论证缺少中间环节，需要增加一节过渡" |
| 阶段5 → 阶段2 | 文献补搜 | 某论点缺乏文献支撑、引用数量不足（<3篇）、关键领域文献缺失 | "关于XX方法的对比实验，仅有1篇文献支撑，需要补充5篇以上" |
| 阶段4 → 阶段2 | 文献补搜 | 文献综述不足以支撑某章节设计、核心参考文献缺失 | "文献综述中缺少XX方向的研究，无法设计相关章节" |

#### 回退执行规则

1. **回退版本管理**：回退后的输出文件加版本号后缀
   - 大纲：`outline_v2.md`、`outline_v3.md`
   - 文献综述：`research/literature_review_v2.md`
   - BibTeX：`research/references_v2.bib`
   - 章节摘要：`chapters/chapter_{N}_summary_v2.md`

2. **回退后章节处理**：
   - 回退点之前的已写章节保留不变
   - 回退点及之后的已写章节标记为 `chapters/chapter_{N}_OBSOLETE.docx`（不删除，留作参考）
   - 从回退点开始重新逐章写作

3. **回退次数限制**：
   - 阶段5→阶段4（大纲修正）：最多 1 次
   - 阶段5→阶段2（文献补搜）：最多 1 次
   - 阶段4→阶段2（文献补搜）：最多 1 次
   - 同一问题方向累计最多 2 次回退
   - 超出限制时：记录问题到 `rollback_feedback.md`，继续推进，在阶段6润色时人工介入

4. **回退后智能体重用规则**：
   - 回退到的阶段必须调用全新智能体实例
   - 新智能体必须读取 `rollback_feedback.md` 获取问题上下文
   - 新智能体必须读取原版本输出文件作为基础（如 `outline.md` → 修正为 `outline_v2.md`）

#### 人工审核断点说明

用户可选择在哪些阶段暂停，等待人工审核确认后再继续。多选，可选任意组合：

| 断点 | 暂停位置 | 审核内容 | 适用场景 |
|------|----------|----------|----------|
| `review_literature` | 阶段2完成后 | 审核搜索到的文献是否覆盖关键方向、是否需要补搜 | 对文献质量要求高、有特定参考文献需求 |
| `review_outline` | 阶段4完成后 | 审核大纲结构是否合理、章节划分是否符合预期 | 对论文结构有明确想法、需要把关论证逻辑 |
| `review_per_chapter` | 阶段5每章完成后 | 逐章审核内容质量、论证充分性、图表是否满意 | 精雕细琢型、对内容质量要求极高 |
| `review_draft` | 阶段5全部完成后 | 审核完整初稿，决定是否需要修改再进入润色 | 大多数正式论文的合理断点 |

**断点触发时的行为：**
- 主智能体在到达断点时暂停，输出当前成果文件路径，等待用户反馈
- 用户可给出修改意见（如"第二章论证不够充分"、"需要增加XX方面的文献"）
- 主智能体根据用户反馈决定：直接继续 / 局部返修 / 触发回退
- 用户回复"继续"或"通过"后，推进到下一阶段

**默认行为（不设断点）：** 全自动执行，仅在回退机制检测到问题时才暂停通知用户。

## 关键约束

### 通用约束
1. **全新实例**：每次执行新任务时，必须调用全新的智能体
2. **无上下文污染**：智能体只返回文件路径，不返回其他信息
3. **最多2轮**：单页审核失败最多重试2次
4. **文件路径通信**：所有智能体之间通过文件路径通信

### PPT工作流额外约束
5. **设计一致性**：PPT必须遵循设计规范，使用统一的组件库
6. **阶段顺序**：必须按顺序执行6个阶段（阶段0可按需执行）
7. **逐页审核**：每一页都必须通过审核（评分>=7.0）才能进入下一页
8. **PPTBuilder专用**：素材智能体必须使用PPTBuilder API，禁止手动创建形状/颜色
9. **素材库优先**：优先使用已提取的组件和配色，禁止手写素材库已有的组件
10. **⚠️ 背景图一致**：除封面和致谢页外，所有页面必须使用相同的bg_type="content"
11. **⚠️ 风格全局一致**：配色、字体、卡片样式在整个PPT中保持统一
12. **⚠️ 布局模式库参考**：内容智能体必须参考布局模式库选择最佳布局
13. **⚠️ 布局必须多样**：禁止连续两页使用相同布局方法
14. **⚠️ 图片表达力>>文字**：能用图绝不用字，每页必须有视觉焦点
15. **⚠️ 占位符强制**：布局规范中必须列出图片占位符，代码中必须生成对应占位矩形
16. **⚠️ 双格式输出**：最终必须同时输出 .pptx 和 .html 两种格式（HTML基于Reveal.js，自包含单文件）

### 论文工作流额外约束
16. **语言一致**：全文语言必须与 requirements.md 中指定的语言一致
17. **引用规范**：每个论点必须有文献支撑，正文使用上标序号式引用 [1]、[2,3]、[4-6]，文末附按序号排列的完整参考文献列表
18. **目录要求**：论文必须包含目录页，位于摘要之后、正文之前，使用 Word TOC 域代码自动生成
19. **逐章审核**：每章写作完成后必须通过质量检查才能进入下一章
20. **文献追踪**：文献搜索必须包含引用关系追踪，不能只做关键词搜索
21. **⚠️ 审稿模式确定**：阶段1必须明确审稿模式（真实/模拟），不可跳过
22. **⚠️ PPT可选**：阶段8为可选阶段，由用户在阶段1决定
23. **⚠️ 输出双格式**：最终必须同时输出 .docx 和 .pdf 两种格式
24. **⚠️ 图表规范**：所有图表必须 ≥300 DPI，PNG 格式，包含完整标注
25. **⚠️ 图表编号**：图表编号格式为 [图X-Y] / [表X-Y]（X=章节号，Y=序号），图注在图下方居中，表注在表上方居中
26. **⚠️ 图表必须在写作阶段内联插入**：图表必须在阶段5逐章写作时就生成并嵌入到 chapter_N.docx 中（使用 run.add_picture() / doc.add_table()），图表必须紧跟在首次引用该图表的段落之后。禁止事后批量插入（会导致位置错误）。每章生成后必须验证 inline_shapes > 0 且 tables > 0
27. **⚠️ 页码规范**：封面无页码，摘要/目录使用罗马数字页码（i, ii, iii...），正文使用阿拉伯数字页码（1, 2, 3...）从1开始
28. **⚠️ BibTeX完整**：references.bib 必须包含所有被引用的文献，格式正确可解析
29. **⚠️ 引用上标拆分**：引用标记 [N] 必须作为独立 run 并设置 superscript=True，禁止混在正文同一个 run 中（否则上标不生效）
30. **⚠️ 参考文献编号不上标**：参考文献列表中每条的 [N] 编号必须是正常文本，禁止设置 superscript
31. **⚠️ 图片嵌入验证**：插入图片后必须验证 doc.part.rels 中存在 image part，且 inline_shapes 数量 > 0，否则图片无法显示
32. **⚠️ 标题样式与分页**：章节标题必须使用 Heading 1 样式（非 Normal），每个章节标题前必须有分页符
33. **⚠️ Word域代码更新**：已封装为脚本 `~/.claude/agents/assets/components/word_com_finalize.py`，润色智能体直接调用即可，不需要自己写 Word COM 代码
34. **⚠️ 分节符和页码处理**：已封装为脚本 `word_com_finalize.py`，自动处理分节符插入、页码添加、域代码更新、中文路径中转
35. **⚠️ 章节合并**：已封装为脚本 `~/.claude/agents/assets/components/merge_docx_chapters.py`，润色智能体直接调用即可，不需要自己写合并代码
36. **⚠️ 章节文件排序必须按数字**：禁止使用字符串排序 `sorted(glob("chapter_*.docx"))`（chapter_10 会排在 chapter_2 前面）。必须按章节号数字排序：`sorted(files, key=lambda f: int(re.search(r'chapter_(\d+)', f).group(1)))`
37. **⚠️ 章节摘要传递机制**：每章写作完成后必须生成 `chapter_{N}_summary.md`（含核心论点、关键术语、结尾铺垫、已用引用编号）。第2章起的内容写作智能体必须读取前一章摘要和正文，确保论点衔接、避免重复论述、保持术语一致。这是解决"全新实例无上下文"导致逻辑断裂的关键机制
38. **⚠️ 回退机制必须启用**：论文工作流不是严格单向的。阶段5（内容写作）和阶段4（架构设计）必须在写作前执行回退检测，发现问题时输出 `rollback_feedback.md` 触发回退，而不是强行继续
39. **⚠️ 回退信号格式统一**：回退信号必须使用 `rollback_feedback.md` 文件，包含回退类型、目标阶段、问题描述、缺失信息、影响范围、建议修改六个字段
40. **⚠️ 回退次数限制**：阶段5→阶段4（大纲修正）最多1次，阶段5→阶段2（文献补搜）最多1次，阶段4→阶段2（文献补搜）最多1次。同一方向累计最多2次回退。超出限制时记录问题继续推进，在阶段6润色时人工介入
41. **⚠️ 回退后版本管理**：回退后的输出文件必须加版本号后缀（outline_v2.md、literature_review_v2.md 等）。回退点之前的已写章节保留，回退点及之后的章节标记为 `_OBSOLETE.docx`，从回退点重新写作
42. **⚠️ 回退后读取最新版本**：所有智能体在读取大纲、文献综述、BibTeX 等可能有多个版本的文件时，必须检查是否存在新版本（如 outline_v2.md），读取最新的版本
43. **⚠️ 人工审核断点必须检查**：主智能体在阶段2、阶段4、阶段5（每章及全部完成后）必须检查 `requirements.md` 中的断点设置。命中断点时必须暂停并等待用户反馈，禁止自动跳过
44. **⚠️ 断点暂停输出规范**：断点暂停时必须输出当前成果文件路径、内容摘要、可执行的操作选项（继续/修改/回退），不得只输出"请审核"而无上下文
45. **⚠️ 逐章格式检查**：阶段5每章写作完成后，必须调用格式检查智能体（paper_format_checker）进行格式检查。检查不通过的章节必须返修后重新检查（最多2轮）
46. **⚠️ 润色智能体职责限定**：润色智能体只负责语言润色和引用上标修复，不负责格式检查（已在阶段5完成）、不负责章节合并脚本编写（调用 merge_docx_chapters.py）、不负责 Word COM 操作（调用 word_com_finalize.py）
